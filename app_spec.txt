<project_specification>
  <project_name>毛叽 Maoji — 宠物健康管家（微信小程序 V1.0 · uni-app 版）</project_name>

  <overview>
    Build the 毛叽 Maoji WeChat Mini Program described in the original PRD
    《毛叽 Maoji · 小程序 MVP V1.0 需求清单（uni-app 版）》, faithfully and
    natively this time: a uni-app (Vue 3 + TS) project that compiles to a
    real WeChat Mini Program (mp-weixin), backed by 微信云开发 (cloud
    functions + cloud database).

    Core value is the **健康主线 (health backbone)**: pet archive +
    vaccination/deworming/medical/medicine/weight records + a unified
    health calendar + automated 订阅消息 reminders + a knowledge library +
    a paid health-membership tier.

    **AI 写真 (AI portrait) is OUT OF SCOPE for V1.0.** Do NOT build the
    style gallery, image generation, AI ordering, or 豆包即梦 integration.
    The 玩法 tab ships only a "敬请期待 (coming soon)" PLACEHOLDER page so the
    4-tab navigation stays intact; everything AI-related is deferred to V2.

    Brand: 毛叽 Maoji (renamed in V1.2 from "毛孩子健康管家").
    Slogan: "毛叽一声，健康同行"
    Audience: 25-40-year-old urban pet owners (cats/dogs), first-time
    owners prioritized.

    ===================================================================
    CRITICAL — DUAL BUILD TARGET (read before writing any code)
    ===================================================================
    The ship target is WeChat Mini Program (mp-weixin). BUT the autonomous
    harness verifies every feature with a HEADLESS-CHROME SCREENSHOT, and a
    mini program cannot run in a browser. uni-app solves this: ONE codebase
    compiles to BOTH targets.

      - DEV / VERIFY loop  -> `npm run dev:h5` -> uni-app H5 build served on
        http://localhost:5173 -> harness drives it with Puppeteer and
        screenshots it. THIS is how features are proven `passes: true`.
      - SHIP target        -> `npm run build:mp-weixin` -> dist/build/mp-weixin
        -> opened in WeChat DevTools, uploaded, submitted for review.

    Therefore EVERY feature MUST render and be verifiable in the H5 build.
    Anything that only exists in mp-weixin (real wx.cloud, real
    requestSubscribeMessage, real WeChat Pay) MUST sit behind an adapter
    that has a working H5 mock, so the H5 build is fully clickable and
    screenshot-able end to end. A feature that "only works in the WeChat
    simulator" is NOT considered passing.
  </overview>

  <technology_stack>
    <frontend>
      <framework>uni-app with Vue 3 (Composition API, &lt;script setup&gt;) + Vite + TypeScript</framework>
      <ui_library>uView Plus (uni-app mobile UI library). Use uView Plus components for all UI; do not pull in browser-only libs (no Vant-web, no DOM-only widgets).</ui_library>
      <state_management>Pinia (uni-app preset). Persist via uni.setStorageSync wrapper, NOT raw localStorage.</state_management>
      <routing>uni-app pages.json (pages + tabBar + subPackages). No vue-router.</routing>
      <styling>uni.scss design tokens + scoped SCSS. Use rpx units (mini-program responsive unit), not px, for layout.</styling>
      <charts>For 体重曲线 use a uni-app-compatible chart (qiun-data-charts / uCharts) that renders on both H5 and mp-weixin canvas. Do NOT use echarts DOM mode.</charts>
      <build_targets>H5 (dev/verify on :5173) AND mp-weixin (ship). Both must compile clean.</build_targets>
    </frontend>
    <backend>
      微信云开发 (WeChat Cloud Development): cloud functions under
      cloudfunctions/, cloud database collections, cloud storage for
      images/PDFs. In the H5 verify build this backend is replaced by a
      mock adapter (see platform_adapter). The cloud-function source code
      still ships in the repo (Node.js), unit-testable in isolation, but is
      NOT required to execute during H5 verification.
    </backend>
    <platform_adapter>
      A single src/platform/* layer wraps every capability that differs
      between H5 and mp-weixin. Each capability exports ONE interface with
      two implementations selected via uni-app conditional compilation
      (#ifdef MP-WEIXIN / #ifdef H5) or a runtime flag:

        - data    : pet/health/knowledge/order CRUD
                    mp-weixin -> wx.cloud.callFunction / database
                    H5        -> mock store backed by uni.setStorageSync +
                      IndexedDB for image blobs (seeded so screens are non-empty)
        - auth    : login + openid
                    mp-weixin -> uni.login -> cloud user-init -> openid
                    H5        -> deterministic demo openid + "Demo 一键登录"
        - subscribeMessage : 健康提醒 templates
                    mp-weixin -> uni.requestSubscribeMessage
                    H5        -> mock that records intent + shows a toast
        - pay     : VIP membership purchase
                    mp-weixin -> uni.requestPayment (via pay-order cloud fn)
                    H5        -> mock success/fail flow
        - storage : image/PDF upload
                    mp-weixin -> wx.cloud.uploadFile
                    H5        -> IndexedDB blob, object-URL preview

      RULE: components/stores NEVER call wx.* or uni.cloud.* directly, and
      NEVER touch localStorage/IndexedDB directly — always go through this
      adapter. This keeps the H5 build runnable and the mp build shippable
      from one source tree.
    </platform_adapter>
    <out_of_scope>
      <item>社区 / UGC feed (晒娃、广场)</item>
      <item>宠物医院 / 商家入驻</item>
      <item>短视频写真</item>
      <item>跨平台同步 / 原生 App build</item>
      <item>AI 健康咨询问答 (V2 重点)</item>
      <item>AI 写真全部功能 (风格商城/上传生成/付费/历史/豆包即梦) — V2，V1.0 仅留占位入口</item>
    </out_of_scope>
  </technology_stack>

  <verification_protocol>
    - init.sh MUST install deps and run `npm run dev:h5`, serving the H5
      build at http://localhost:5173.
    - The harness verifies each feature by navigating the H5 build with
      Puppeteer and screenshotting; a feature is `passes: true` ONLY after a
      screenshot (or console-log/vitest output) has been Read as evidence.
    - ALSO verify the project still compiles for the ship target:
      `npm run build:mp-weixin` must succeed without errors (run it at least
      once per session; capture the build log as evidence). A feature that
      breaks the mp-weixin compile is NOT passing even if H5 looks fine.
    - Pure logic (healthCycle, cycle math, date utils) verified via vitest.
    - NEVER use wx.* or localStorage directly in .vue/.ts outside platform/*.
  </verification_protocol>

  <uniapp_structure_requirements>
    These are HARD, separately-verifiable acceptance gates. The initializer
    MUST emit explicit feature_list.json test cases for EACH of them, and
    they must be provable by grep/file-inspection or screenshot — not left
    as implicit assumptions.

    1. pages.json is the single routing source:
       - Exists at src/pages.json; valid JSON.
       - "pages" array registers EVERY page under pages/ (index, pet/list,
         pet/detail, pet/edit, health/{vaccine,deworm,medical,medicine,
         weight,calendar}, knowledge/{list,detail}, user/{profile,vip,orders})
         and the 玩法 placeholder.
       - "tabBar" declares EXACTLY 4 items in order 首页/档案/玩法/我的, each
         pagePath pointing at a registered page.
       - NO vue-router anywhere (grep: zero "vue-router" / "createRouter").

    2. manifest.json is a real uni-app manifest:
       - Exists at src/manifest.json; valid JSON.
       - "vueVersion": "3".
       - Has an "mp-weixin" section with an "appid" field (placeholder
         "touristappid" or a TODO is fine) so build:mp-weixin can run.
       - "h5" section present so dev:h5 serves on port 5173.

    3. UI is built with uView Plus components, NOT browser libs:
       - uView Plus is a dependency in package.json and registered in main.ts
         (app.use(uviewPlus) or easycom configured in pages.json).
       - Real screens use u-* components (u-button, u-cell, u-form, u-popup,
         u-tabs, u-icon, etc.) for their primary widgets.
       - grep proves ZERO usage of Vant ("vant"), element-plus, or other
         web-only UI kits anywhere in src/.

    4. Mini-program-correct primitives (grep-verifiable):
       - Layout uses rpx units, not px, for sizing/spacing in styles.
       - Charts use uCharts / qiun-data-charts on <canvas>, NOT DOM echarts
         (grep: zero "echarts" import in components).
       - Storage/persistence goes through uni.setStorageSync wrappers inside
         src/platform/*, never raw localStorage.

    A build/UI that renders fine in H5 but FAILS any of 1-4 is NOT passing:
    it would not survive compilation to / behave correctly as a real WeChat
    Mini Program, which is the actual ship target.
  </uniapp_structure_requirements>
